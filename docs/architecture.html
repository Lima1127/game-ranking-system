🎮 Game Ranking System

Documentação Técnica Detalhada

📋 Visão Geral Técnica

O Game Ranking System é uma aplicação backend desenvolvida em Spring Boot 3.3.5 com Java 21 que implementa um sistema de ranking competitivo para jogadores. A aplicação oferece APIs RESTful para gerenciar usuários, edições de competição, jogos, conclusões e um mecanismo sofisticado de pontuação.

🎯 ScoringEngine - Motor de Pontuação

O ScoringEngine é o componente responsável por analisar cada Completion e gerar ScoreEvents correspondentes. É um componente crítico do sistema que implementa toda a lógica de recompensas.

Critérios de Pontuação

🎮 Jogo Completado 1 ponto
🆕 Primeira Experiência - Primeira vez na vida completando este jogo 1 ponto
🥇 Primeiro na Edição - Primeiro participante a completar nesta edição 1 ponto
📅 Em Dia - Completou no mesmo ano de lançamento 1 ponto
⏱️ Tempo Valioso (bloco) - A cada 25 horas jogadas 2 pontos/bloco
👑 Platina - Atingiu 100% de completude 3 pontos
🤝 Braço Direito - Completou em coop (até 4 jogadores) 2 pontos
🔥 Participação no Hype - Participou de eventos especiais 1 ponto

Algoritmo de Exemplo

function buildCompletionEvents(completion, edition, user, underdogBonus):
    events = []
    
    // Base
    events.add(GAME_COMPLETED, 1 ponto)
    
    // Histórico
    if completion.firstTimeEver:
        events.add(FIRST_EXPERIENCE, 1 ponto)
    
    // Competição
    if completion.firstInEdition:
        events.add(FIRST_IN_EDITION, 1 ponto)
    
    // Timing
    if completion.completedInReleaseYear:
        events.add(IN_RELEASE_YEAR, 1 ponto)
    
    // Tempo investido (blocos de 25h)
    blocks = floor(completion.hoursPlayed / 25)
    for i in 0..blocks:
        events.add(TIME_VALUABLE_BLOCK, 2 pontos)
    
    // Completude
    if completion.platinum:
        events.add(PLATINUM, 3 pontos)
    
    // Modo cooperativo
    if completion.coop AND completion.coopPlayers <= 4:
        events.add(COOP_RIGHT_HAND, 2 pontos)
    
    // Evento especial
    if completion.hyped:
        events.add(HYPE_PARTICIPATION, 1 ponto)
    
    return events

🔄 Fluxo de Dados Principal

1. Fluxo de Autenticação

POST /api/v1/auth/register ↓ AuthController.register() ↓ AuthService.register() ├─ Valida email (não existe) ├─ Hash de senha (bcrypt) ├─ Cria User (ativo) └─ UserRepository.save() ↓ Retorna AuthResponse com ├─ token (JWT) └─ expiresIn (7200s = 2h)

2. Fluxo de Registro de Conclusão

POST /api/v1/completions Header: X-User-Id: {userId} Body: CreateCompletionRequest ↓ CompletionController.create() ├─ Valida X-User-Id header └─ Chama CompletionService.create(userId, request) ↓ CompletionService.create() ├─ Carrega User, Game, Edition ├─ Valida período de edição ├─ Cria Completion entity ├─ CompletionRepository.save() └─ Chama ScoringEngine ↓ ScoringEngine.buildCompletionEvents() ├─ Analisa cada critério ├─ Gera lista de ScoreEvent └─ ScoreEventRepository.saveAll() ↓ Retorna CompletionResponse com ├─ Id da conclusão ├─ Pontos totais └─ Lista de eventos

3. Fluxo de Consulta de Ranking

GET /api/v1/ranking?editionId={editionId} ↓ RankingController.get() └─ Chama RankingService.getRanking(editionId) ↓ RankingService.getRanking() ├─ Se editionId vazio, busca edição ativa ├─ Query native SQL: │ SELECT user_id, displayName, SUM(points) as totalPoints │ FROM score_events │ WHERE edition_id = ? │ GROUP BY user_id, displayName │ ORDER BY totalPoints DESC └─ Mapeia resultado para RankingRowResponse ↓ Retorna List<RankingRowResponse> ├─ Posição ├─ Usuario (displayName) └─ Pontos totais

4. Estrutura de Segurança com JWT

🔐 Token JWT
Estrutura: Header.Payload.Signature

Payload contém:
• sub: Email do usuário (subject)
• iat: Issued at (timestamp)
• exp: Expiration (timestamp + 7200s)
• role: Papel do usuário (ADMIN/USER)

Validação:
• Assinatura HMAC-SHA256 com secret configurável
• Expiração validada a cada request
• Header X-User-Id extrai userId do contexto seguro

⚡ Otimizações e Performance

Query Optimization

  • Eager Loading: Carrega Game e User na Completion para evitar N+1
  • Native Queries: RankingService usa SQL nativa para agregações
  • Índices: Criados em FK (user_id, game_id, edition_id)
  • Pagination: Implementada para listagens grandes
